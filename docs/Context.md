# Pet App V3 开发上下文记录

## 📋 项目概述

Pet App V3 是一个基于插件化架构的多平台应用框架，采用"万物皆插件"的设计理念，支持动态插件加载、热重载、权限管理等企业级功能。

**当前版本**: v4.3.0
**开发状态**: Phase 4.3 已完成 ✅，桌宠系统核心实现完成
**下一阶段**: Phase 5 高级功能特性 (第三方集成、云服务支持)

## 🤖 **AI智能助手愿景**

### **桌宠增强理念**
```
桌宠 = 悬浮陪伴 + AI助手 + 系统集成者
├── 悬浮窗常驻 (类似快捷球，跨平台支持)
├── 自然语言交互 (NLP意图理解，多轮对话)
├── 深度模块集成 (调用Pet App各功能模块)
├── LLM智能对话 (多模型支持，本地+云端)
└── 终端集成 (代码辅助，如Gemini CLI插件)
```

### **应用场景示例**
```
场景1: "进入川策通,搜索关于四川地区医保的报销范围"
├── NLP解析: 目标模块(川策通) + 操作(搜索) + 内容(医保报销)
├── 模块调用: 启动川策通插件
├── RAG检索: 调用模块内LLM进行知识检索
└── 结果展示: 智能整理并展示搜索结果

场景2: "帮我写一个Python爬虫脚本"
├── NLP解析: 任务类型(编程) + 语言(Python) + 功能(爬虫)
├── 终端调用: 启动Termux或集成终端
├── LLM辅助: 生成代码框架和实现建议
└── 交互编程: 用户与AI协作完成代码
```

### **技术架构参考**
基于Claude Code逆向分析的核心设计：
```
1. 实时Steering机制 (h2A双重缓冲异步消息队列)
   - 零延迟消息传递，吞吐量 > 10,000 消息/秒
   - Promise-based异步迭代器 + 智能背压控制

2. 分层多Agent架构
   - 主Agent (nO主循环引擎) - 核心任务调度
   - SubAgent (I2A子任务代理) - 隔离执行环境
   - Task Agent (专用任务处理器) - 并发执行

3. 智能上下文管理
   - 92%阈值自动压缩，wU2压缩器智能保留关键信息
   - CLAUDE.md长期记忆存储，动态上下文窗口调整

4. 工具执行框架
   - 6阶段执行管道，最大10并发控制
   - 完整权限验证系统，沙箱隔离执行
```

#### **审查结果统计**
| 优先级 | 问题数量 | 影响范围 |
|--------|----------|----------|

| **总计** | **18个** | **整体功能受限** |

#### **Critical 优先级问题 (阻塞核心功能)**
1. **WorkshopManager核心占位符**
   - `_registerBuiltinTools()` - 完全占位符，无工具注册
   - `_registerBuiltinGames()` - 完全占位符，无游戏注册
   - `_loadUserToolPlugin()` - 无动态加载逻辑
   - `_loadUserGamePlugin()` - 无动态加载逻辑

2. **工具插件架构不匹配**
   - `SimpleBrushTool`、`SimplePencilTool` 继承`ChangeNotifier`而非`ToolPlugin`
   - 无法集成到Plugin System的企业级架构

3. **游戏插件架构不匹配**
   - `SimpleGame`、`SimpleClickGame` 继承`ChangeNotifier`而非`GamePlugin`
   - 无法集成到Plugin System的企业级架构

#### **High 优先级问题 (影响用户体验)**
1. **Web平台存储缺失** - `LocalProjectStorage`的Web实现全是占位符
2. **用户插件管理** - `_getUserPluginList()`使用硬编码模拟数据
3. **UI功能不完整** - 工具栏清空画布、工具激活等功能为占位符

#### **Medium 优先级问题 (影响功能完整性)**
1. **形状工具缺失** - `_showShapeTools()`仅显示提示消息
2. **撤销重做功能** - `_HistoryManager`缺少实际操作记录
